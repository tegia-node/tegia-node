# Task-004: Базовая валидация JSON в `ADD_WS_ACTION_ROUTE`

- Статус: Выполнено
- Дата: 2026-03-26
- Основание: `docs/adr/ADR-0002-ws-route-registration-model.md`
- Цель: добавить базовую проверку `route_json` в `ADD_WS_ACTION_ROUTE`, чтобы рано ловить опечатки в ключах (`action`/`actor`/`mapping`).

## 1. Контекст

После перехода на новую сигнатуру `ADD_WS_ACTION_ROUTE` ошибка в ключе JSON (`"actin"` вместо `"action"`) приводит к падению при старте. Нужна предсказуемая и явная валидация с понятным сообщением об ошибке.

## 2. Объем работ

Входит:
1. Добавить базовую валидацию `route_json` внутри `ADD_WS_ACTION_ROUTE`.
2. В случае ошибки — логировать понятное сообщение и завершать процесс (fail-fast), как и в других проверках маршрутов.
3. Обновить документацию с указанием обязательных полей JSON.

Не входит:
1. Изменение структуры `route_json` (контракт остается прежним).
2. Изменение поведения `ADD_WS_ROUTE`.

## 3. Системная аналитика

### 3.1 Точки входа и место проверки

Сейчас `ADD_WS_ACTION_ROUTE` напрямую читает `route_json["action"]` и вызывает `get<std::string>()`. При отсутствии ключа — исключение из `nlohmann::json`. Логика должна быть симметрична проверкам в `type_base_t::add_route`, но происходить раньше, в момент декларации.

### 3.2 Требования к валидации

Минимальный набор проверок:
1. `route_json` — объект.
2. `route_json["actor"]` — строка.
3. `route_json["action"]` — строка и начинается с `/`.
4. `route_json["mapping"]` — существует (тип валидируется текущей логикой `add_route`).

### 3.3 Поведение при ошибке

- Логировать ошибку с кодом `WS_ROUTE_JSON_INVALID` через `std::cout`.
- В лог включить `type`, `method`, `pattern`, `route_json.dump()` и причину.
- Завершить процесс `exit(0)` (fail-fast), как и в других проверках маршрутов.

### 3.4 Риски

- Дублирование валидации между `ADD_WS_ACTION_ROUTE` и `type_base_t::add_route`.
- Необходимо убедиться, что сообщения об ошибке читаемы и не замыливают логи.
- Валидация ограничена обязательными ключами; глубинная проверка схемы остается на `add_route`.

## 4. Точки изменения по файлам

1. `include/tegia/actors/type.h` — добавить проверку `route_json` в `ADD_WS_ACTION_ROUTE`.
2. `include/tegia/docs/ws-routing-contract.md` — явно перечислить обязательные ключи `route_json`.
3. `include/tegia/docs/quickstart-new-actor.md` — добавить короткое примечание про обязательные ключи.

## 5. Критерии приемки

1. При отсутствии `actor/action/mapping` — четкая ошибка в логах и fail-fast на старте.
2. Опечатка в ключе (`actin`) приводит к диагностируемой ошибке, а не к JSON исключению.
3. Документация отражает обязательность полей `actor/action/mapping`.

## 6. Решения

1. Код события: `WS_ROUTE_JSON_INVALID`.
2. Логирование ошибки — через `std::cout` с последующим `exit(0)`.
3. Валидация — только обязательные ключи (`actor`, `action`, `mapping`) и базовые типы. Глубинная проверка схемы остается в `add_route`.
