# Task-006: Механизм самовыгрузки WS‑актора при ошибке инициализации

- Статус: Запланировано
- Дата: 2026-03-30
- Основание: технический долг из `docs/tasks/task-005.md`
- Цель: определить и реализовать механизм корректной выгрузки WS‑актора из памяти при неуспешной инициализации `data`.

## 1. Контекст

При ошибке инициализации `data` (например, в `/init`) запись в БД не создается, но актор остается в памяти. В текущем runtime отсутствует механизм самовыгрузки, что оставляет «подвисшие» WS‑акторы.

## 2. Объем работ

Входит:
1. Спроектировать механизм завершения/выгрузки WS‑актора при неуспешной инициализации.
2. Определить точки вызова (например, при ошибке `init_data(...)`).
3. Зафиксировать поведение в документации и/или ADR, если требуется.

Не входит:
1. Изменение контракта `/init`.
2. Рефакторинг логики инициализации `data` (это в Task-005).

## 3. Критерии приемки

1. При ошибке инициализации `data` WS‑актор корректно выгружается из памяти.
2. Ошибка логируется в `L3ERROR`.
3. Поведение не приводит к утечкам ресурсов (`_router`, `_system`, DB‑handles).

## 4. Открытые вопросы

1. Какой механизм выгрузки допустим в текущем runtime (явное уничтожение, сигнал в диспетчер, отложенная очистка)?
2. Кто инициирует выгрузку: сам актор или диспетчер/супервизор?
