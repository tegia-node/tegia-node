# Task-005: Реализация ADR-0004 — виртуальные `set_data/get_data` в `ws_t`

- Статус: Запланировано
- Дата: 2026-03-30
- Основание: `docs/adr/ADR-0004-ws-data-initialization.md`
- Цель: внедрить виртуальные методы `set_data/get_data` в `ws_t` для инициализации и сериализации `data` без изменения иерархии.

## 1. Контекст

ADR-0004 фиксирует решение: `ws_t` остается базовым классом WS, а работа с пользовательскими данными `data` выполняется через виртуальные методы `set_data/get_data`, которые обязаны переопределять наследники.

## 2. Объем работ

Входит:
1. Добавить в `ws_t` виртуальные методы `set_data/get_data`.
2. Обновить `ws_t::init()` и `ws_t::commit()` для использования `set_data/get_data`.
3. Адаптировать существующие WS (наследники `ws_t`) для переопределения `set_data/get_data`.
4. Обновить внутреннюю документацию по использованию `set_data/get_data`.

Не входит:
1. Изменение модели `system`.
2. Пересмотр протокола `/init`, если не требуется по ходу внедрения.

## 3. Детали реализации

1. Валидация `/init`:
   - `ws_t` валидирует базовую структуру запроса и обязательные системные поля;
   - `set_data(...)` валидирует и инициализирует `data` из полного `message->data`.
2. Формат сериализации `data` не меняется: используется поле `data` в БД.
3. Структура БД не меняется; миграции данных выполняются внутри `set_data(...)` при необходимости.
4. Потокобезопасность и жизненный цикл `data` определяются в реализации наследника и не меняются в рамках этой задачи.

## 3.0 Последовательность /init

При вызове маршрута `/init` выполняется следующая последовательность:
1. Создание/инициализация `_system`.
2. Инициализация `data` через виртуальный метод `set_data(message->data)`.
3. Сохранение записи о WS в БД (включая поле `data`).

## 3.1 Предлагаемые интерфейсы

Ниже фиксируются минимальные интерфейсы классов как точка согласования для реализации.

```cpp
// Базовый класс WS.
class ws_t : public actor_t {
public:
    virtual ~ws_t() = default;

    // Базовая логика WS (перенос из текущего ws_t).
    unsigned long long int roles(const std::string& uuid);
    int commit();
    int router(const std::shared_ptr<message_t>& message);
    int init(const std::shared_ptr<message_t>& message);
    int member_add(const std::shared_ptr<message_t>& message);
    int member_accept(const std::shared_ptr<message_t>& message);
    int member_remove(const std::shared_ptr<message_t>& message);

protected:
    ws_t(
        const std::string& type,
        const std::string& name,
        const std::string& connection,
        const std::string& table,
        unsigned long long int creators);

    // Базовые внутренние операции.
    int load_type_routes();
    int _create();

    // Обязательный контракт наследников для data.
    virtual int set_data(const json& data) = 0;
    virtual json get_data() const = 0;

    // Доступ к системным данным WS.
    tegia::ws::system_t* _system = nullptr;
    tegia::app::router_t* _router = nullptr;

    std::string connection;
    std::string table;
    std::string ws_title;
    unsigned long long int creators;

    // Хранение сырого data из БД (опционально для наследников).
    json _data_raw;
};

// Пример наследника.
class my_ws_t : public ws_t {
protected:
    int set_data(const json& data) override;
    json get_data() const override;
};
```

Примечания:
1. `set_data/get_data` обязательны для всех наследников `ws_t`.
2. `get_data()` используется в `commit()` для записи `data` в БД.

## 3.2 Перенос методов и полей из текущего `ws_t`

Основано на текущей реализации `ws_t` в `include/tegia/ws/ws.h` и `src/ws/ws.cpp`.

### Остается в `ws_t` (базовая логика WS)

Методы:
1. `ws_t::ws_t(...)` — базовая инициализация WS (создание `_router`, `_system`, загрузка из БД, `load_type_routes`, проверка прав создателя).
2. `ws_t::~ws_t()` — освобождение `_router`, `_system`.
3. `int ws_t::commit()` — базовая фиксация WS (без сохранения типизированной `data`).
4. `unsigned long long int ws_t::roles(...)`.
5. `int ws_t::load_type_routes()`.
6. `int ws_t::_create()`.
7. `int ws_t::router(...)`.
8. `int ws_t::init(...)`.
9. `int ws_t::member_add(...)`.
10. `int ws_t::member_accept(...)`.
11. `int ws_t::member_remove(...)`.

Поля:
1. `_router`, `_system`.
2. `connection`, `table`, `ws_title`, `creators`.
3. Все базовые поля из `actor_t`, используемые логикой WS (`type`, `name`, `status`, `ws`).

### Добавить в наследники `ws_t`

Методы:
1. `int set_data(const json& data)` — инициализация/валидация пользовательских данных.
2. `json get_data() const` — сериализация данных для записи в БД.

### Дополнительно

1. `type_t<actor_type>` остается без изменений (базовый тип `ws_t` сохраняется).

## 4. Точки изменения по файлам

1. `include/tegia/...` — объявление `ws_base_t` и `ws_t<DataT>`.
2. `src/...` — реализация базовой логики WS в `ws_base_t`.
3. `docs/adr/ADR-0004-ws-data-initialization.md` — при необходимости уточнения решения.
4. `docs/...` — обновление внутренней документации (runtime).

## 5. Критерии приемки

1. `/init` вызывает `set_data(...)`, при ошибке WS не создается.
2. `commit()` сохраняет `data` через `get_data()`.
3. Коллекции WS продолжают работать через базовый `ws_t` без изменений.
4. Производительность вне `set_data/get_data` не ухудшается.
5. Существующие WS мигрированы без изменения внешнего поведения.

## 6. Закрытые вопросы и решения

1. Контракт ошибок при `set_data(...)`:
   - коды ошибок: `200` — данные успешно инициализированы, `400` — данные невалидны, `500` — внутренняя ошибка;
   - логирование через `L3ERROR`;
   - запись в БД не создается, откат не требуется;
   - актор остается в памяти (нет механизма самовыгрузки) — зафиксировано как технический долг в отдельной задаче.
   - обработка кодов — вызывающей стороной, базовый маршрут `/init`.

2. `type_t<actor_type>` и регистрация экшенов:
   - изменений не требуется, базовый тип `ws_t` сохраняется.

3. План тестирования (обязательные проверки):
   - создание нового WS и базовая инициализация (`_system`, `load_type_routes`, создание записи в БД);
   - `/init` с валидными данными `message->data` (успешная инициализация `data` и сохранение);
   - `/init` с невалидными данными (ошибка `init_data`, логирование `L3ERROR`, запись в БД не создается);
   - загрузка существующего WS из БД (корректная загрузка `data` и `system`);
   - маршрутизация `/router` и доступ к ролям участников (`roles`, `member_*`).
   - тестирование выполняется вручную на тестовом стенде.

## 7. Статус готовности

Задача не готова к реализации: требуется зафиксировать обязательные детали реализации для `set_data/get_data`.

## 8. Дополнительные уточнения (принятые решения)

1. `set_data(...)` вызывается внутри `ws_t::init(...)` (маршрут `/init`).
2. Загрузка `data` при создании WS из БД выполняется через `set_data(_data_raw)` в конструкторе наследника (при необходимости).
3. `get_data()` возвращает JSON для записи в БД, формат — JSON.
4. Формат ошибок в HTTP‑ответе не фиксируется; достаточно внутренних кодов возврата.
5. Повторная инициализация запрещена: `set_data(...)` вызывается только если WS еще не инициализировано.

## 9. Замечания и открытые вопросы (аудит)

3. Не описано, как именно `commit()` использует `get_data()` и что происходит при ошибке сериализации.
