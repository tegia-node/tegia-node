# ADR-0004: Инициализация и хранение `data` WS на уровне базового класса

- Статус: Черновик
- Дата: 2026-03-30
- Автор: system architect
- Область: `tegia-node` runtime, `ws_t`, `/init`, модель данных WS

## Контекст

На уровне WS существует два типа данных:
1. `system` — внутренние данные WS, задаются в базовом классе `ws_t` и недоступны в наследниках.
2. `data` — пользовательские данные, доступные участникам WS.

Проблема: при вызове `/init` в момент создания WS приходят пользовательские данные, но объект `data` определяется в классах-наследниках и не может быть инициализирован на уровне базового класса. Это создает разрыв между местом получения данных (`ws_t`) и местом их определения (наследник).

## Драйверы решения

1. Инициализация `data` должна происходить в момент создания WS, на уровне обработки `/init`.
2. Структура `data` должна определяться в наследниках, без ручной прокладки через базовый класс.
3. Базовый класс должен иметь доступ к `data` для общих операций (логирование, сериализация, общие проверки).
4. Решение должно быть совместимо с хранением коллекций WS и общих механизмов, которые опираются на базовый тип.
5. Минимизировать риски регрессий и удержать производительность на уровне текущего runtime.

## Варианты

### Вариант 1: шаблонный базовый класс (CRTP)
Сделать `ws_t` шаблонным по `DataT` и хранить `data` как `DataT` в базовом классе.

### Вариант 2: нешаблонный `ws_base_t` + шаблонный `ws_t<DataT>`
Выделить `ws_base_t` с полной базовой логикой WS. Наследник `ws_t<DataT>` отвечает только за работу с `data` по стандартизированному интерфейсу (CRTP).

### Вариант 3: виртуальная фабрика + type-erasure
`ws_t` хранит `std::any`/`std::unique_ptr<base_data_t>`, а наследник создает и инициализирует `data` через виртуальный метод.

### Вариант 4: интерфейс инициализации + коллбек
`ws_t` получает коллбек (инициализатор `data`), зарегистрированный наследником.

### Вариант 5: PIMPL-подход для `data`
`ws_t` хранит `std::unique_ptr<data_iface_t>`; конкретная реализация задается в наследнике.

### Вариант 6: смешанная модель
`ws_t` хранит `data_iface_t`, а наследник предоставляет типизированный доступ.

### Вариант 7: виртуальные `set_data/get_data` в `ws_t`
Оставить `ws_t` нешаблонным и ввести виртуальные методы `set_data(const json&)` и `get_data() const`, которые обязаны переопределять наследники.

## Критерии выбора и оценка

### Критерии и шкала

Шкала: 1 — плохо, 3 — приемлемо, 5 — отлично.

1. Типобезопасность при работе с `data`.
2. Простота интеграции с текущим кодом `ws_t`.
3. Удобство работы с коллекциями WS и общими механизмами (`ws_base`).
4. Стоимость миграции и риск регрессий.
5. Сложность сопровождения и отладки.
6. Производительность (накладные расходы на вызовы и хранение `data`).

### Пояснения к оценкам

**Вариант 1: шаблонный базовый класс (CRTP)**
- Типобезопасность: 5 — `data` статически типизирована в базовом классе.
- Интеграция: 2 — базовый класс становится шаблонным, затрагивает всю иерархию.
- Коллекции: 1 — без type-erasure сложно хранить в общих контейнерах.
- Миграция: 1 — высокий риск регрессий и большой рефакторинг.
- Сопровождение: 3 — шаблоны усложняют отладку, но типизация чистая.
- Производительность: 5 — прямой доступ без виртуальных вызовов.

**Вариант 2: `ws_base_t` + `ws_t<DataT>`**
- Типобезопасность: 5 — `data` типизирована в `ws_t<DataT>`.
- Интеграция: 3 — требуется выделить `ws_base_t`, но остальная логика остается нетипизированной.
- Коллекции: 5 — общий код работает с `ws_base_t`.
- Миграция: 3 — нужен рефакторинг, но локализованный.
- Сопровождение: 4 — шаблонная часть тонкая и изолированная.
- Производительность: 5 — `data` хранится без type-erasure.

**Вариант 3: виртуальная фабрика + type-erasure**
- Типобезопасность: 2 — тип контролируется только в рантайме.
- Интеграция: 5 — минимум изменений в текущей иерархии.
- Коллекции: 5 — единый базовый тип сохраняется.
- Миграция: 5 — минимальная стоимость.
- Сопровождение: 2 — риск ошибок при приведениях.
- Производительность: 2 — виртуальные вызовы и type-erasure.

**Вариант 4: интерфейс инициализации + коллбек**
- Типобезопасность: 3 — зависит от типа хранения, обычно нужна проверка в рантайме.
- Интеграция: 5 — минимальные изменения, простой контракт инициализации.
- Коллекции: 5 — `ws_t` остается нешаблонным.
- Миграция: 5 — низкая стоимость.
- Сопровождение: 2 — риск роста скрытых зависимостей через коллбеки.
- Производительность: 3 — индирекция при инициализации и доступе.

**Вариант 5: PIMPL-подход для `data`**
- Типобезопасность: 2 — тип скрыт за интерфейсом.
- Интеграция: 3 — требуется новый интерфейс `data`, но базовый класс остается.
- Коллекции: 5 — единый базовый тип сохраняется.
- Миграция: 3 — средняя стоимость рефакторинга.
- Сопровождение: 2 — сложнее трассировать ошибки.
- Производительность: 2 — аллокации и виртуальные вызовы.

**Вариант 6: смешанная модель**
- Типобезопасность: 3 — типизировано в наследнике, но базовый класс опирается на interface/type-erasure.
- Интеграция: 3 — требует нового интерфейса и соглашений.
- Коллекции: 5 — базовый класс не шаблонный.
- Миграция: 3 — умеренные изменения.
- Сопровождение: 2 — дисциплинарные риски и ошибки приведения.
- Производительность: 3 — компромисс за счет type-erasure в базовом классе.

**Вариант 7: виртуальные `set_data/get_data`**
- Типобезопасность: 2 — все взаимодействие через `json`, проверка в рантайме.
- Интеграция: 5 — минимальные изменения, `ws_t` остается базовым классом.
- Коллекции: 5 — единый базовый тип сохраняется.
- Миграция: 5 — локальные изменения в наследниках.
- Сопровождение: 4 — простой контракт двух методов.
- Производительность: 3 — накладные расходы только на вызов `virtual`.

## Решение

Выбран **Вариант 7: виртуальные `set_data/get_data` в `ws_t`**.

Ключевые причины выбора:
1. Минимальные изменения в иерархии и регистрации действий.
2. Сохранение единого базового типа `ws_t` и простоты работы с коллекциями.
3. Накладные расходы ограничены только вызовами `virtual` методов `set_data/get_data`.
4. Быстрая миграция существующих WS — достаточно переопределить два метода.

## Детали решения

1. `ws_t` остается нешаблонным базовым классом WS.
2. В `ws_t` вводятся два обязательных виртуальных метода:
   - `int set_data(const json& data)` — инициализация/валидация пользовательских данных;
   - `json get_data() const` — сериализация данных для записи в БД.
3. Маршрут `/init` вызывает `set_data(message->data)`. При ошибке инициализации WS не создается.
4. `commit()` использует `get_data()` для сохранения поля `data` в БД.
5. Валидация `/init`, сериализация, миграции и потокобезопасность остаются ответственностью наследников.

## Совместимость

- Миграция сводится к переопределению `set_data/get_data` в наследниках `ws_t`.
- Формат `/init` сохраняется, изменения протокола не требуются.

## Риски и ограничения

1. Потеря статической типизации `data` на уровне `ws_t`.
2. Ошибки валидации выявляются только в рантайме.
3. Небольшая накладная стоимость вызова `virtual` методов.

## План внедрения (черновик)

1. Добавить виртуальные `set_data/get_data` в `ws_t`.
2. Обновить `ws_t::init()` и `ws_t::commit()` для использования `set_data/get_data`.
3. Адаптировать все наследники `ws_t` (реализовать два метода).
4. Проверить интеграцию `/init` и корректность инициализации `data`.
5. Обновить внутреннюю документацию.

## Критерии приемки

1. `/init` вызывает `set_data(...)`, при ошибке WS не создается.
2. `commit()` сохраняет `data` через `get_data()`.
3. Коллекции WS продолжают работать с базовым `ws_t` без изменений.
4. Производительность не ниже текущего уровня для операций вне `set_data/get_data`.

## Закрытые вопросы

1. Интерфейс инициализации `data` — виртуальные `set_data/get_data` в `ws_t`.
2. Остальные вопросы (валидация `/init`, сериализация, миграции, потокобезопасность) переведены в задачи внедрения.
