# ADR-0003: Проверка прав доступа на уровне WS route

- Статус: Черновик
- Дата: 2026-03-26
- Автор: system architect
- Область: `tegia-node` runtime, `router_t`, `ws_t`, контракт route

## Контекст

Сейчас права доступа проверяются в `tegia::actors::map_t::action_func` по `action->roles`, т.е. после резолва маршрута. Роутер (`router_t`) не учитывает роли и пропускает любой корректно сматченный route.

Проблемы текущего подхода:
- нет возможности отфильтровать запрос на уровне маршрута до передачи управления action;
- для внешних actor (другой node) локального `action` может не быть, но маршрут все равно принимается;
- семантика доступа не закреплена в контракте route.

## Драйверы решения

1. Единые и явные правила доступа для локальных и внешних actor.
2. Раннее отклонение запросов без передачи управления в action.
3. Совместимость с текущей моделью `ADD_ACTION` и `ADD_WS_ACTION_ROUTE`.
4. Возможность декларативного контроля доступа для внешних маршрутов.

## Варианты

### Вариант A: оставить как есть
Проверка прав только в `action_func` по `action->roles`.

Плюсы:
- нулевая стоимость изменений;
- минимальный риск регрессий.

Минусы:
- нет раннего отказа на уровне маршрута;
- для внешних actor отсутствует декларативная политика доступа.

### Вариант B: route-level как предварительный фильтр
Добавить `route.roles` и проверять его в `router_t::match(...)` или на входе в `ws_t::router(...)`. Итоговая проверка остается в `action_func` по `action->roles`.

Плюсы:
- ранний отказ;
- не ломает текущую модель `action->roles`.

Минусы:
- два источника прав (route + action);
- нужна политика согласования.

### Вариант C: route-level как источник истины
Перенести контроль доступа на уровень route (route.roles), а `action->roles` использовать только для локальных вызовов без HTTP (внутренние сообщения).

Плюсы:
- единый источник истины для HTTP маршрутов.

Минусы:
- ломает текущую модель безопасности;
- усложняет поддержку локальных action и несогласованных путей вызова.

## Решение (предпочтительный вариант)

**Вариант B**: ввести `route.roles` как предварительный фильтр с сохранением финальной проверки в `action_func`.

Правила:
1. `route.roles` задается для каждого WS маршрута.
2. Если `route.roles` отсутствует:
   - для маршрутов, созданных через `ADD_WS_ACTION_ROUTE`, значение берется из `action->roles` (на этапе регистрации route);
   - для маршрутов, созданных через `ADD_WS_ROUTE`, отсутствие `route.roles` считается ошибкой загрузки типа.
3. `router_t` применяет `route.roles` как early-check до создания сообщения.
4. `action_func` сохраняет текущую проверку `action->roles` (defense-in-depth).

## Детали реализации (эскиз)

1. Расширить структуру `route_t`/`route.data` полем `roles`.
2. В `ADD_WS_ACTION_ROUTE` автоматически добавлять `roles` в `route.data`.
3. В `ADD_WS_ROUTE` требовать `roles` явно (в `data_json`).
4. В `router_t::match(...)` перед маппингом данных проверять `roles`:
   - если не совпало — вернуть ошибку доступа до dispatch в action.
5. Для внешних actor (когда `actor` не `{/this}`) `route.roles` является единственным источником доступа.

## Совместимость

- Для legacy instance-level `this->_router->add(...)` при отсутствии `roles` использовать поведение по умолчанию (reject или allow) должно быть определено отдельным флагом. Рекомендуется: fail-fast на старте для новых типов, но оставить legacy без изменения в фазе миграции.

## Риски и ограничения

- Дублирование проверки (route + action) может вводить рассогласование, если роли не синхронизированы.
- Нужны явные правила при конфликте `route.roles` и `action->roles` (например: route.roles — подмножество action.roles).

## План внедрения (черновик)

1. Ввести `route.roles` в контракте и API регистрации.
2. Автоматически наполнять `route.roles` в `ADD_WS_ACTION_ROUTE`.
3. Добавить проверку в `router_t::match(...)`.
4. Документировать новый контракт.
5. Добавить тесты: разрешен/запрещен на уровне route + согласование с action.

## Критерии приемки

1. При отсутствии прав запрос не проходит маршрутизацию (ранний отказ).
2. Для `ADD_WS_ACTION_ROUTE` роли в route совпадают с `action->roles`.
3. Для `ADD_WS_ROUTE` отсутствие `roles` вызывает ошибку загрузки типа.
4. Существующие WS-акторы не ломаются в фазе миграции (legacy).
